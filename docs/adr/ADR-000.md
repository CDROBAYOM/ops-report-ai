# ADR-000 — Decisión fundacional del proyecto

**Título:** Alcance y límites del sistema de reportes operativos  
**Estado:** Aceptado  
**Fecha:** Junio 2026  
**Autor:** Carlos David Robayo  

---

## Contexto

Las pymes gestionan su operación entre Excel compartidos, mensajes de WhatsApp y correos con archivos adjuntos. El resultado es una ausencia de fuente única de verdad: cada área trabaja con su propia versión de los datos, nadie es formalmente responsable de su exactitud, y las decisiones gerenciales se toman con información desactualizada o inconsistente.

Ante este problema, la respuesta más común en la industria es inmediata: adoptar una herramienta de Business Intelligence, implementar un ERP, o delegar el análisis a un modelo de inteligencia artificial.

Este proyecto rechaza ese camino como punto de partida.

La razón es simple: **ninguna herramienta puede definir qué dato importa si el negocio no lo ha definido primero.** Automatizar sobre una base mal definida produce el mismo error con mayor velocidad.

---

## Decisión

Este sistema se construye sobre una premisa no negociable:

> **El diseño del modelo de datos, la definición de métricas y la asignación de responsabilidades sobre cada dato son decisiones del negocio — no del sistema, no de la IA, no del arquitecto.**

En consecuencia, el proyecto establece los siguientes límites explícitos:

### Lo que este sistema hace

- Ingesta datos desde fuentes heterogéneas: archivos CSV, Excel y webhooks
- Normaliza y estandariza la información recibida siguiendo reglas definidas por el negocio
- Clasifica y agrupa registros según categorías previamente acordadas con el equipo operativo
- Genera reportes consolidados de forma automática y periódica
- Usa IA para asistir en la normalización de datos inconsistentes o con formato variable
- Alerta cuando un dato esperado no llegó o presenta anomalías respecto al histórico

### Lo que este sistema deliberadamente NO hace

- **No decide qué métricas son importantes** — eso lo define el negocio con el arquitecto antes de escribir una línea de código
- **No valida si un dato es correcto desde el punto de vista del negocio** — solo detecta inconsistencias técnicas
- **No reemplaza al responsable del dato** — el sistema registra quién es el dueño de cada métrica, pero no suplanta ese rol
- **No genera conclusiones de negocio autónomamente** — resume y presenta, no interpreta ni recomienda
- **No se adapta solo a cambios en el negocio** — si cambia una métrica clave, ese cambio pasa primero por una decisión documentada

## El paso previo al sistema — no delegable

Antes de configurar cualquier componente de este proyecto, deben estar respondidas por escrito las siguientes preguntas con el equipo del negocio:

| Pregunta | Por qué no puede delegarse a la IA |
|---|---|
| ¿Para qué decisión específica necesita este reporte el negocio? | La IA no conoce el contexto estratégico del negocio |
| ¿Quién es el responsable nombrado de cada dato? | La responsabilidad es un acuerdo humano, no técnico |
| ¿Con qué frecuencia debe actualizarse cada métrica? | Depende del ritmo de decisión del negocio, no del sistema |
| ¿Qué significa una anomalía en este contexto? | El umbral de alerta lo define quien conoce la operación |
| ¿Cómo sabremos que el problema mejoró? | La métrica de éxito debe existir antes de empezar |

Si alguna de estas preguntas no tiene respuesta clara, **el desarrollo no debe comenzar.**

## Consecuencias de esta decisión

**Positivas:**

- El sistema resuelve el problema real — no automatiza el caos, lo ordena
- Cada componente construido tiene una justificación de negocio documentada
- El límite entre responsabilidad humana y responsabilidad del sistema es explícito y trazable
- La IA opera dentro de un alcance acotado, auditado y reversible

**Negativas y compromisos asumidos:**

- El proyecto requiere una fase de discovery con el negocio antes de cualquier desarrollo — eso agrega tiempo inicial
- No es un sistema plug-and-play — cada implementación requiere configuración basada en decisiones del negocio
- La calidad del output depende directamente de la calidad del trabajo previo al código

---

## Alternativas consideradas y descartadas

**Delegar la definición de métricas a la IA mediante análisis exploratorio de los datos existentes**
Descartada. Un modelo puede identificar patrones en datos históricos, pero no puede saber qué decisiones gerenciales dependen de qué indicadores. Eso requiere contexto de negocio que no está en los datos.

**Comenzar con un modelo de datos genérico y ajustarlo iterativamente**
Descartada como punto de partida. Un modelo genérico genera reportes que nadie usa porque no responden las preguntas reales del negocio. La iteración es válida, pero sobre una base de decisiones conscientes — no sobre supuestos.

**Implementar primero la infraestructura técnica y luego definir el modelo de negocio**
Descartada. Construir la solución antes de entender el problema es exactamente el error que este proyecto existe para demostrar cómo evitar.

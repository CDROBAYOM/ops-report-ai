# ADR-003 — Alcance y límites del uso de IA en el pipeline

**Título:** La IA asiste la normalización — no toma decisiones de negocio
**Estado:** Aceptado
**Fecha:** Junio 2026
**Autor:** Carlos David Robayo
**Referencia:** ADR-000, ADR-002

---

## Contexto

Este proyecto existe para demostrar que la IA es una herramienta de asistencia,
no un sustituto del criterio humano ni de las reglas del negocio.

El pipeline tiene un paso específico donde la variabilidad del dato no puede
resolverse de forma determinística: las fechas con formato ambiguo.

Cuando el archivo de Andrés contiene `05/06/2025`, el sistema no puede saber
con certeza si es el 5 de junio o el 6 de mayo sin contexto adicional.
Ese es el único problema en este sistema que justifica asistencia de IA.

Todo lo demás — detectar columnas, eliminar duplicados, calcular el índice
de mora, generar la alerta — tiene una solución determinística y no necesita IA.

---

## Decisión

La IA opera en un único paso del pipeline, con entrada acotada,
salida verificable y sin capacidad de modificar entidades del dominio.

### Alcance exacto — solo normalización de fechas ambiguas

```
Entrada al servicio de IA:
  - El valor crudo de la fecha: "05/06/2025"
  - El contexto del archivo: otras fechas del mismo archivo
  - El período esperado: semana de reporte (conocido antes del proceso)

Salida esperada del servicio de IA:
  - La fecha en formato ISO 8601: "2025-06-05"
  - Nivel de confianza: alto | medio | bajo
  - Justificación: por qué interpretó dd/mm y no mm/dd
```

Si el nivel de confianza es **bajo**, el servicio no devuelve una fecha —
devuelve una señal de ambigüedad. El adaptador convierte esa señal en
una `AlertaOperativa(anomalia_dato)` y detiene el proceso.

**La IA nunca escribe directamente en una entidad del dominio.**
Su output es siempre un dato primitivo (string de fecha) que el adaptador
valida antes de pasarlo al caso de uso.

### Lo que la IA NO hace en este sistema

| Acción | Por qué está prohibida |
|---|---|
| Decidir si el índice de mora es aceptable | Es una regla del negocio definida en el dominio |
| Inferir valores para campos vacíos | Introduce incertidumbre no trazable en el reporte |
| Clasificar un crédito como "en mora" o "al día" | Depende de reglas contractuales del negocio |
| Generar o modificar una `AlertaOperativa` | Es una entidad del dominio — solo el caso de uso la crea |
| Decidir si un archivo es válido o no | Es responsabilidad del adaptador (ADR-002) |
| Resumir o interpretar el reporte final | Don Carlos interpreta — el sistema reporta |

---

## Por qué este límite es una decisión de arquitectura

Permitir que la IA opere fuera de este alcance acotado introduce dos riesgos
que este proyecto existe para evidenciar:

**Riesgo 1 — Trazabilidad perdida**
Si la IA infiere un valor para `Saldo_Vigente` vacío y ese valor es incorrecto,
el índice de mora calculado es incorrecto. Don Carlos activa el comité de riesgo
por un dato que el sistema inventó. Ese error no es auditable.

**Riesgo 2 — Responsabilidad difusa**
Cuando la IA toma decisiones de negocio, nadie puede responder quién decidió.
En una cooperativa regulada, cada decisión sobre cartera en mora debe tener
un responsable humano identificable. La IA no puede ser ese responsable.

## Implementación del servicio

```typescript
// src/infrastructure/ai/normalizacion.service.ts

interface FechaNormalizada {
  valor: string;           // ISO 8601 si hay confianza suficiente
  confianza: 'alta' | 'media' | 'baja';
  justificacion: string;
}

interface NormalizacionService {
  normalizarFecha(
    valorCrudo: string,
    contexto: string[]    // otras fechas del mismo archivo
  ): Promise<FechaNormalizada>;
}
```

El servicio es una interfaz — el caso de uso no sabe si la implementación
usa un modelo de lenguaje, una librería de parsing, o una tabla de reglas.
Eso puede cambiar sin tocar el dominio.

---

## Regla de expansión futura

Si en el futuro se identifica un nuevo caso donde la IA podría asistir,
la decisión de incorporarla debe seguir este criterio:

1. ¿El problema tiene una solución determinística? → No usar IA
2. ¿El output de la IA es verificable antes de llegar al dominio? → Puede considerarse
3. ¿Un error de la IA en este paso tiene impacto en una decisión de negocio? → Requiere aprobación explícita documentada en un nuevo ADR

---

## Alternativas consideradas y descartadas

**Usar IA para normalizar todos los problemas del archivo (columnas, duplicados, fechas)**
Descartada. Los problemas de columnas y duplicados tienen solución
determinística documentada en ADR-002. Usar IA donde no es necesario
aumenta la superficie de error y reduce la trazabilidad sin aportar valor.

**No usar IA en ningún paso — resolver fechas con reglas fijas**
Descartada para el caso de fechas genuinamente ambiguas. Una regla fija
(`asumir siempre dd/mm`) introduce errores silenciosos cuando el archivo
mezcla formatos en el mismo campo. La IA con nivel de confianza explícito
es más honesta sobre su incertidumbre que una regla que nunca falla en silencio.

**Usar IA para generar el resumen ejecutivo del reporte**
Descartada. Generar conclusiones sobre la cartera es exactamente el tipo
de decisión que este proyecto demuestra que no debe delegarse a la IA.
Don Carlos necesita el dato — la interpretación es su trabajo.

---

## Consecuencias de esta decisión

**Positivas:**
- Cada decisión de negocio tiene un responsable humano identificable
- El output de la IA es verificable y auditable antes de llegar al dominio
- El sistema puede operar sin IA si el servicio no está disponible —
  genera una alerta y detiene el proceso en lugar de fallar silenciosamente

**Negativas y compromisos asumidos:**
- El servicio de normalización de fechas debe mantenerse cuando aparecen
  nuevos formatos no contemplados
- La confianza "media" requiere una política explícita — ¿se procesa o se detiene?
  Esa política debe definirla CoopNorte, no el sistema

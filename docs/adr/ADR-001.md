# ADR-001 — Arquitectura hexagonal como estructura del sistema

**Título:** Adopción de arquitectura hexagonal (Ports & Adapters)
**Estado:** Aceptado
**Fecha:** Junio 2026
**Autor:** Carlos David Robayo
**Referencia:** ADR-000 — Decisión fundacional del proyecto

---

## Contexto

El sistema recibe datos desde fuentes heterogéneas con formatos inconsistentes
(Excel con columnas variables, fechas en formatos distintos, registros duplicados)
y debe producir dos entidades de dominio bien definidas: `ReporteCartera` y
`AlertaOperativa`.

El caso de CoopNorte tiene una característica crítica: **las fuentes de entrada
pueden cambiar en cualquier momento.** Hoy es un Excel exportado desde SisCredito.
Mañana puede ser un CSV, una API REST o un webhook. El dominio del negocio —
qué es un índice de mora, qué constituye una alerta — no debería cambiar cuando
cambia la fuente.

Adicionalmente, el sistema usa IA para asistir en la normalización de datos.
Esa dependencia externa tampoco debe contaminar la lógica de negocio.

---

## Decisión

El sistema se estructura en tres capas explícitas siguiendo el patrón
**Ports & Adapters (Hexagonal Architecture)**:

```
┌─────────────────────────────────────────────────────┐
│                   INFRAESTRUCTURA                   │
│  ┌─────────────┐              ┌──────────────────┐  │
│  │  Adaptador  │              │    Adaptador     │  │
│  │  de Entrada │              │    de Salida     │  │
│  │  (Excel/CSV)│              │  (Reporte/Alerta)│  │
│  └──────┬──────┘              └────────┬─────────┘  │
│         │                             │             │
│  ───────┼─────────────────────────────┼──────────   │
│         │         APLICACIÓN          │             │
│  ┌──────▼──────────────────────────────▼─────────┐  │
│  │         ProcesarReporteCartera (UseCase)       │  │
│  └──────────────────────┬────────────────────────┘  │
│  ───────────────────────┼──────────────────────────  │
│         │            DOMINIO          │             │
│  ┌──────▼─────────────────────────────────────────┐  │
│  │  ReporteCartera          AlertaOperativa        │  │
│  │  (entidad)               (entidad)              │  │
│  └────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### Las tres capas y su responsabilidad

**Dominio** — el corazón del sistema. Contiene las entidades
`ReporteCartera` y `AlertaOperativa` y las reglas de negocio puras:
qué es un índice de mora, cuándo se genera una alerta, qué campos
son obligatorios. Esta capa no conoce Excel, no conoce IA, no conoce
bases de datos. Solo conoce el negocio de CoopNorte.

**Aplicación** — orquesta el flujo. El caso de uso
`ProcesarReporteCartera` coordina: recibir los datos normalizados,
construir el `ReporteCartera`, evaluar si corresponde una
`AlertaOperativa`, y devolver el resultado. No sabe de dónde vienen
los datos ni adónde van.

**Infraestructura** — todo lo que cambia. Los adaptadores de entrada
(lectura del Excel, normalización con IA) y los adaptadores de salida
(persistencia del reporte, emisión de alertas). Esta capa conoce
openpyxl, conoce la API de IA, conoce la base de datos. El dominio
no sabe que ella existe.

    │   └── output/

## Por qué esta decisión para este caso específico

**Protege el dominio de los cambios en la fuente.**
Cuando Andrés cambie el formato del Excel — y lo hará, ya lo hace cada semana —
solo cambia el adaptador de entrada. Las entidades `ReporteCartera` y
`AlertaOperativa` no se tocan.

**Acota el alcance de la IA.**
El servicio de IA vive exclusivamente en `infrastructure/ai/`. No puede
acceder al dominio directamente. Eso garantiza que la IA asiste en la
normalización pero nunca toca la lógica de negocio — lo que el ADR-000
establece como límite no negociable.

**Hace los tests independientes de la infraestructura.**
Los tests del dominio y del caso de uso no necesitan un archivo Excel real
ni una API de IA activa. Eso es lo que permite usar los fixtures de
`test/fixtures/` de forma controlada.

---

## Consecuencias de esta decisión

**Positivas:**
- Cambiar la fuente de datos no requiere tocar el dominio
- La IA puede reemplazarse o desactivarse sin afectar las entidades
- Los tests unitarios del dominio son rápidos y sin dependencias externas
- La separación de capas hace el código legible para alguien que no lo escribió

**Negativas y compromisos asumidos:**
- Más archivos y carpetas que una solución directa script → output
- Requiere disciplina para no romper las fronteras entre capas
- Para un caso simple, puede parecer sobreingeniería — este proyecto
  existe precisamente para demostrar que no lo es cuando el problema
  lo justifica

---

## Alternativas consideradas y descartadas

**Script lineal: leer Excel → procesar → escribir resultado**
Descartada. Funciona para una semana. Cuando cambia el formato del Excel
en la semana siguiente, el script se rompe en el mismo lugar donde
está la lógica de negocio. No hay separación posible.

**Arquitectura en capas tradicional (MVC)**
Descartada. El controlador termina conociendo detalles de la fuente de datos
y la lógica de negocio queda distribuida entre servicios sin una frontera clara.
Para este caso, la dependencia en la fuente de datos es el riesgo principal —
hexagonal lo resuelve mejor.

> y la lógica de negocio debe sobrevivir a ese caos.*

# ADR-002 — Estrategia de ingesta desde fuentes heterogéneas

**Título:** Ingesta de archivos Excel con formato variable mediante adaptador tolerante
**Estado:** Aceptado
**Fecha:** Junio 2026
**Autor:** Carlos David Robayo
**Referencia:** ADR-001 — Arquitectura hexagonal

---

## Contexto

El archivo fuente de CoopNorte (exportación de SisCredito v2.1) presenta
tres problemas técnicos documentados en los fixtures de prueba:

**Problema 1 — Nombres de columna inconsistentes** (`cartera_semana_02.xlsx`)
El mismo campo llega con nombres distintos dependiendo de quién exportó:
`Valor_Mora`, `valor mora`, `MORA`, `Valor_Cartera_Vencida`.

**Problema 2 — Formatos de fecha mezclados** (`cartera_semana_01.xlsx`)
En el mismo archivo coexisten `dd/mm/yyyy` y `mm-dd-yyyy` sin patrón predecible.

**Problema 3 — Registros duplicados y celdas vacías** (`cartera_semana_03.xlsx`)
Un crédito con múltiples cuotas vencidas puede aparecer más de una vez.
Campos obligatorios como `Saldo_Vigente` pueden llegar vacíos.

La pregunta de diseño es: ¿dónde se resuelven estos problemas y cómo?

---

## Decisión

El adaptador de ingesta opera en dos fases secuenciales con responsabilidades
estrictamente separadas. Si cualquier validación falla, el proceso se detiene
y se genera una `AlertaOperativa` de tipo `anomalia_dato` — nunca se procesa
un archivo con datos en estado incierto.

### Fase 1 — Detección y mapeo de columnas

El adaptador no asume nombres de columna fijos. Mantiene un mapa de aliases
conocidos por campo del dominio:

```typescript
const COLUMN_ALIASES = {
  saldoVigente: [
    'Saldo_Vigente', 'SALDO ACTUAL', 'saldo actual',
    'Saldo_Cartera', 'SALDO_VIGENTE'
  ],
  valorMora: [
    'Valor_Mora', 'valor mora', 'MORA',
    'Valor_Cartera_Vencida', 'VALOR_MORA'
  ],
  fechaDesembolso: [
    'Fecha_Desembolso', 'Fecha_Otorgamiento',
    'FECHA_DESEMBOLSO', 'Fecha_Inicio'
  ]
}
```

Si ningún alias coincide con las columnas del archivo recibido, el adaptador
no intenta adivinar — genera una `AlertaOperativa` y detiene el proceso.

**Este mapa de aliases es una decisión del negocio, no del adaptador.**
Cada nuevo alias que se agregue debe estar justificado por una exportación
real del sistema fuente — no por suposición técnica.

### Fase 2 — Validación antes de normalización

Antes de pasar los datos al servicio de normalización, el adaptador valida:

| Validación | Acción si falla |
|---|---|
| Columnas requeridas presentes | `AlertaOperativa(anomalia_dato)` + detener |
| Sin registros duplicados por `ID_Credito` | Eliminar duplicados + registrar en log |
| Campos monetarios sin símbolo `$` ni comas | Limpiar y convertir a número |
| Fechas parseables en algún formato conocido | Delegar a servicio de normalización (ADR-003) |
| `Saldo_Vigente` no nulo en registros activos | `AlertaOperativa(anomalia_dato)` + detener |

**Regla crítica:** los duplicados se eliminan silenciosamente pero se registran.
Los campos obligatorios vacíos detienen el proceso — no se asume ningún valor.

### Lo que el adaptador NO hace

- No corrige datos que no entiende — los marca y detiene
- No infiere valores para campos vacíos
- No decide si un formato de fecha ambiguo es `dd/mm` o `mm/dd` —
  eso lo resuelve el servicio de normalización con IA (ADR-003)
- No modifica el archivo fuente original — solo lo lee

---

## Flujo completo del adaptador

```
Recibir archivo Excel
        ↓
Detectar columnas → ¿todas presentes?
   NO  → AlertaOperativa(anomalia_dato) → fin
   SÍ  ↓
Eliminar duplicados → registrar en log
        ↓
Validar campos obligatorios → ¿alguno vacío?
   SÍ  → AlertaOperativa(anomalia_dato) → fin
   NO  ↓
Limpiar valores monetarios
        ↓
Pasar a servicio de normalización (ADR-003)
        ↓
Retornar datos listos para el caso de uso
```

---

## Alternativas consideradas y descartadas

**Asumir nombres de columna fijos y fallar si cambian**
Descartada. El sistema fuente es legacy y no está bajo nuestro control.
Asumir estabilidad en algo que sabemos que cambia es una deuda técnica
que explota en producción.

**Usar IA para detectar las columnas correctas automáticamente**
Descartada. Delegar la detección de columnas a la IA introduce
incertidumbre en un paso que puede y debe ser determinístico.
Si la IA se equivoca al mapear `valor mora` → `saldoVigente`,
el índice de mora es incorrecto y don Carlos toma una decisión errónea.
El costo de ese error supera el costo de mantener el mapa de aliases.

**Rechazar cualquier archivo que no cumpla exactamente el formato esperado**
Descartada. CoopNorte no tiene capacidad técnica para estandarizar
sus exportaciones en el corto plazo. El sistema debe ser tolerante
a la variabilidad conocida sin ser permisivo con la ambigüedad.

---

## Consecuencias de esta decisión

**Positivas:**
- El proceso nunca produce un reporte con datos en estado incierto
- El mapa de aliases es un artefacto vivo que documenta la variabilidad
  real del sistema fuente
- Los errores son trazables — cada archivo procesado deja un registro
  de qué se limpió y qué se detuvo

**Negativas y compromisos asumidos:**
- El mapa de aliases debe mantenerse cuando aparece un nuevo formato
- Detener el proceso ante campos vacíos puede generar falsas alarmas
  si el sistema fuente envía campos opcionales sin valor — ese caso
  debe documentarse como excepción explícita cuando ocurra
